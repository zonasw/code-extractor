# CodeExtractor 开发者使用指南

这份文档面向日常写代码的开发者，帮你把 CodeExtractor 真正融入工作流——不只是"把文件复制给 AI"，而是用 Agent 模式高效完成新增功能、修 bug、重构、写测试等真实任务。

---

## Agent 模式：常见任务怎么做

### 新增功能

把相关模块的文件选中（接口定义、已有的相似功能实现、类型文件），然后直接描述需求：

> "参照 `src/api/posts.ts` 的写法，在 `src/api/comments.ts` 新增一个 `deleteComment(id: string)` 方法，包含错误处理，并更新对应的类型定义。"

**建议**：新功能开发用**目录模式**——让 Agent 自己去读相关文件，省得你费心挑选。

### 修 bug

把出错的文件和错误日志一起塞给 Agent：

> "运行时报错如下：`TypeError: Cannot read properties of undefined (reading 'map')` 出现在 `UserList` 组件里。帮我找出根本原因并修复，不要引入 async 状态管理的副作用。"

**建议**：用**内联模式**，把报错的组件和它依赖的 hook 直接选中，让 Agent 有完整上下文。

### 重构

重构最怕 Agent 自作聪明改太多。先用**仅规划模式**看方案，确认没问题再执行：

> "把 `src/components/DataTable` 里所有直接操作 DOM 的地方改成 React ref，其他逻辑不动。给我一个改动计划，不要直接修改文件。"

看完计划觉得OK，再切换到正常模式重新跑一遍。

### 写测试

选中要测试的文件，告诉 Agent 你用的测试框架：

> "为 `src/utils/formatters.ts` 里的所有导出函数写 Vitest 单元测试，覆盖正常路径和边界 case（空值、超长字符串、非法输入）。测试文件放在 `src/utils/__tests__/formatters.test.ts`。"

## Context 选择策略：内联 vs 目录

### 内联模式

把你选中的文件内容**直接嵌入 prompt**。

**适合的情况：**
- 任务范围明确，相关文件就那几个
- 你希望 Agent 严格聚焦，不要到处乱跑
- 文件总 token 在 50K 以内

**缺点：**
- 选多了超 token 限制，选少了 Agent 缺乏上下文猜出来的结果不对

### 目录模式

只把目录路径传给 Agent，让它用 Read / Glob / Grep 工具自己去探索。

**适合的情况：**
- 大型项目，你不确定哪些文件相关
- 任务涉及多个模块（比如"全局替换某个 API 的调用方式"）
- 你想让 Agent 自己分析依赖链

**缺点：**
- Agent 可能读很多无关文件，耗时更长、花费更高

### 经验法则

| 场景 | 推荐模式 |
|---|---|
| 修一个具体的 bug（文件已知） | 内联 |
| 新增功能，涉及 3 个以上文件 | 目录 |
| 重构整个模块 | 目录 |
| 写某个函数的单测 | 内联 |
| 分析全局影响范围 | 目录 |

**选太多的问题**：噪音变多，Agent 容易走偏，token 费用也高。
**选太少的问题**：Agent 缺乏上下文，容易给出和现有代码风格不符的方案。

## 权限模式详解

### 接受编辑（Accept Edits）

Agent 可以直接读写文件，无需每步确认。适合你对任务比较有把握、改动范围清晰的场景。用完看 diff，不满意就 revert。

### 跳过权限（Skip Permissions）

所有操作自动批准，包括执行 Bash 命令。**只在你完全信任 prompt 内容时使用**，比如跑已知的 lint fix 或格式化。

### 默认（Default）

对敏感操作（如文件写入、执行命令）会暂停请求确认。适合你不确定 Agent 会做什么的情况。

### 仅规划（Plan Only）

**强烈推荐用于重构和大改动。**

Agent 只分析、只输出计划，不修改任何文件。你能看到：
- 哪些文件会被改
- 每个文件改什么
- 改动的先后顺序和逻辑

看完觉得方向对了，再切换到"接受编辑"模式重新运行相同的 prompt 执行。

**典型使用流程：**
1. 仅规划模式 → 看输出的改动计划
2. 如有问题，追加追问（多轮对话）澄清
3. 切换到接受编辑 → 执行
4. 看 diff → 确认或 revert

## 多轮对话最佳实践

### 什么时候继续上一个对话（--resume）

- 上一个任务做了一半，想继续
- 想在上一步结果的基础上追加需求
- Agent 的第一次实现有问题，你想让它自己修

在 Agent 面板的**历史会话**里找到对应的 session，点击继续即可（内部用 `--resume` 标志）。

### 什么时候开一个新对话

- 任务性质完全不同（比如刚写完一个功能，下一个要修另一个模块的 bug）
- 上一轮的改动已经 commit，开始一个全新的需求
- 历史上下文太多可能干扰判断的时候

**原则**：一个对话聚焦一件事。塞太多不同任务的历史上下文会让 Agent 混淆。

### 追问技巧

Agent 给的方案不完全对时，不需要重新开始，直接追问：

> "你刚才的实现在并发场景下有问题，因为 `userCache` 不是线程安全的。请修改这部分，其他已完成的部分不动。"

## Skills（技能模板）：统一团队 Prompt 规范

Skills 是存放在 `~/.claude/commands/*.md` 的 Markdown 文件，在 Agent 面板里可以一键插入到任务输入框。

团队可以把常用的 prompt 规范化，避免每个人写法不一、质量参差不齐。

### 在 CodeExtractor 里管理 Skills

点击 Agent 面板右上角的 **Skills** 按钮，可以：
- 查看现有 skills 列表
- 创建新 skill（直接写 Markdown）
- 编辑已有 skill
- 点击插入到当前任务输入框

### 4 个实用 Skill 示例

**1. 代码审查（`code-review.md`）**

```markdown
# 代码审查

仔细审查我提供的代码，从以下几个维度输出报告：

1. **正确性**：有没有逻辑错误、边界 case 遗漏、空值未处理
2. **安全性**：有没有 SQL 注入、XSS、敏感信息泄露的风险
3. **性能**：有没有不必要的重复计算、内存泄漏、N+1 查询
4. **可维护性**：命名是否清晰、函数职责是否单一、有没有重复代码

每个问题给出：问题描述 + 文件位置 + 修改建议。
严重问题用 🔴，建议改进用 🟡，优化点用 🟢。
```

**2. 写单元测试（`write-tests.md`）**

```markdown
# 写单元测试

为我提供的代码写 Vitest 单元测试，要求：

- 覆盖所有导出的函数/方法
- 每个函数至少包含：正常路径、边界 case、异常/错误路径
- Mock 外部依赖（HTTP 请求、数据库、文件系统）
- 测试描述用中文，清晰说明在测什么
- 测试文件放在 `__tests__/` 目录下，文件名和源文件对应

输出测试文件内容，不要解释，直接给可运行的代码。
```

**3. 生成文档（`gen-docs.md`）**

```markdown
# 生成技术文档

为我提供的代码生成技术文档，包含：

1. **模块概述**：这个模块做什么、解决什么问题
2. **对外接口**：每个导出的函数/类，包含参数说明、返回值、使用示例
3. **内部逻辑**：关键流程的文字说明（不需要逐行解释）
4. **依赖关系**：依赖哪些外部模块/服务

格式用 Markdown，适合直接放进项目 wiki。
```

**4. 分析 Bug（`analyze-bug.md`）**

```markdown
# 分析 Bug

我会提供错误信息和相关代码，请帮我：

1. **定位根本原因**：从错误信息和调用栈找到真正出问题的地方
2. **复现路径**：描述触发这个 bug 的操作序列
3. **影响范围**：这个 bug 还会在哪些其他场景下出现
4. **修复方案**：给出 1-3 个修复思路，分析各自的利弊
5. **预防建议**：怎么避免同类问题再次出现

不要急着给代码，先把分析结论说清楚。
```

## Git 工作流集成

### 标准流程

1. **切功能分支**

   ```bash
   git checkout -b feature/my-feature
   ```

2. **在 CodeExtractor 里跑 Agent**

   Agent 面板顶部会显示当前 git 分支，确认在正确的分支上再跑。

3. **用 diff 检查改动**

   任务完成后，展开 diff 查看器逐文件检查。重点看：
   - Agent 有没有改动它不该碰的文件
   - 逻辑是否符合预期
   - 有没有留下调试代码或 TODO

4. **满意就 commit，不满意就 revert**

   一键 Revert 会执行 `git restore .`，回到 Agent 运行前的状态。然后可以修改 prompt 重新跑。

5. **提 PR 前再跑一遍代码审查 Skill**

   用 `code-review` skill 让 Agent 自审一遍，把明显问题在 PR 前修掉。

### 注意事项

- Agent 开始前如果有未提交的改动，diff 查看器里会混入，建议**先 commit 或 stash 再跑 Agent**。
- 不要在 `main` 或 `master` 分支上直接跑有写权限的 Agent，养成在功能分支上操作的习惯。

## 选择预设：为不同任务创建专属预设

选择预设（Selection Presets）可以把常用的文件组合保存下来，适合你在不同任务之间快速切换。

### 推荐预设方案

**前端组件预设**

包含：`src/components/**`、`src/hooks/**`、`src/styles/**`

用于：UI bug 修复、新增组件、样式调整

**后端 API 预设**

包含：`src/api/**`、`src/types/**`、`src/middleware/**`

用于：新增接口、数据验证、错误处理

**测试相关预设**

包含：需要补测试的业务模块 + 对应的 `__tests__/` 目录

用于：批量补充测试覆盖率

**配置文件预设**

包含：`*.config.*`、`tsconfig.json`、`.env.example`、`package.json`

用于：排查依赖问题、调整构建配置

### 创建预设

1. 在左侧面板选中目标文件
2. 打开 Config 标签页
3. 在"选择预设"区域点击**保存当前选择**，输入预设名称
4. 下次直接从下拉列表一键载入
