# CodeExtractor 产品经理使用指南

这份文档面向产品经理，帮你用 CodeExtractor 在不懂代码的前提下，真正看懂代码库、评估需求工作量、和开发高效对齐。

你不需要会写代码，也不需要有本地开发环境——很多场景用**代码提取模式**把文件内容发给 Claude.ai 就够了。

---

## 产品经理能用 Agent / 代码提取做什么

| 场景 | 推荐用法 |
|---|---|
| 看懂某个功能是怎么实现的 | 提取代码 → 发给 Claude.ai 解释 |
| 评估需求工作量 | Agent 仅规划模式 → 列出改动文件和步骤 |
| 技术债分析 | Agent 仅规划模式 → 代码质量扫描 |
| 生成 Release Notes / Changelog | 提取 diff → 发给 Claude.ai 或 Agent 输出 |
| 快速找到某类文件 | 左侧面板用 Glob 搜索 |

---

## 先了解两种工作模式

### 模式一：代码提取（不需要 Agent）

在左侧面板选中文件 → 复制 → 粘贴到 Claude.ai 或任何 AI 工具里提问。

**适合你的场景**：你只需要读懂代码、生成文档，不需要 AI 修改任何东西。这种方式**不需要配置 API key，不需要安装额外工具**，打开 CodeExtractor 就能用。

### 模式二：Agent（需要配置 API Key）

在 Agent 面板里直接跑，AI 可以探索整个代码库、回答问题、输出分析报告。

**产品经理使用时：始终选择"仅规划"模式**，Agent 只分析不改代码，完全安全。

## 典型场景一：看懂一个功能是怎么实现的

**背景**：你想了解"订单取消"功能的完整逻辑，比如取消后库存是否自动回补、退款是同步还是异步。

**用法 A（提取模式，推荐新手）：**

1. 在左侧搜索框输入 `*cancel*` 或 `*order*`，找到相关文件
2. 选中这些文件，看右下角的 token 计数，确保不超过 200K
3. 点 Config 标签，格式选 **XML**（Claude 读起来最清晰）
4. Ctrl+Shift+C 复制，粘贴到 Claude.ai
5. 提问：

   > "这些是我们订单取消功能的相关代码。请用非技术语言解释：1）取消流程的完整步骤；2）取消后库存和退款是怎么处理的；3）有哪些情况下取消会失败；4）这个功能有什么已知的限制或潜在问题。"

**用法 B（Agent 模式，适合已配置好的用户）：**

1. 在 Agent 面板，权限模式选**仅规划**
2. 输入：

   > "请找到我们代码库里处理订单取消的所有相关逻辑，然后用产品经理能看懂的语言解释完整流程。重点说明：取消触发条件、库存回补机制、退款流程、失败情况的处理。"

3. 运行，读输出

## 典型场景二：评估需求工作量

**背景**：你有一个新需求——"支持批量导出订单为 Excel"，你想知道开发大概要改哪些地方、工作量多大。

**操作步骤：**

1. 在左侧面板，用 Glob 搜索 `*order*` 或 `*export*`，找到相关模块
2. Agent 面板，权限模式选**仅规划**
3. 输入：

   > "我有一个新需求：用户可以在订单列表页批量选择订单并导出为 Excel 文件，Excel 包含订单号、金额、状态、创建时间。
   >
   > 请分析：
   > 1. 实现这个功能大概需要改动哪些文件
   > 2. 每个文件需要做什么改动（简要说明）
   > 3. 有哪些技术风险或依赖（比如需要新引入第三方库）
   > 4. 预估这是一个大改动、中等改动还是小改动，理由是什么
   >
   > 不需要写代码，给我一份改动清单就好。"

**输出可以用来：**

- 在需求评审会上和开发快速对齐范围
- 在 Jira Epic 里写技术分解
- 判断需求是否需要拆期

## 典型场景三：技术债分析

**背景**：你需要给管理层汇报技术债情况，说明为什么某些功能"改起来特别慢"。

**操作步骤：**

1. 选中你认为"最难改"或"bug 最多"的几个模块
2. Agent 面板，权限模式选**仅规划**
3. 输入：

   > "请扫描我选中的这些文件，输出一份技术债评估报告，面向非技术管理层。
   >
   > 报告结构：
   > 1. **总体健康度**：用一句话总结（好/中/差，为什么）
   > 2. **主要问题清单**：每个问题说明是什么、为什么是问题、对业务有什么影响（比如"导致每次改这里都要修 3 个地方"）
   > 3. **优先级建议**：哪些问题应该先处理，为什么
   > 4. **修复成本预估**：大/中/小
   >
   > 不要用太多技术术语，用业务语言描述影响。"

**技巧**：可以选几个已知出过 bug 的模块，让 Agent 解释为什么这些地方容易出问题，这比抽象的技术债描述更有说服力。

## 典型场景四：生成 Release Notes / Changelog

**背景**：版本发布前，你需要写一份面向用户或内部的变更日志，但看 git commit 记录太痛苦了。

**用法 A（提取模式）：**

1. 让开发给你本次版本的 git diff 或改动文件列表
2. 在 CodeExtractor 里选中这些文件
3. 复制，粘贴到 Claude.ai
4. 提问：

   > "这些是本次版本的代码改动。请生成一份面向用户的 Release Notes，格式：
   >
   > ## v1.x.x（日期）
   >
   > ### 新功能
   > - （用户视角描述，不要技术术语）
   >
   > ### 优化
   > -
   >
   > ### 修复
   > -
   >
   > 每条用一句话说清楚，用户能感知到的改变才写，纯内部重构不写。"

**用法 B（Agent 模式）：**

1. Agent 面板输入：

   > "请查看最近的 git 提交记录（git log），整理本次迭代的改动，生成一份面向外部用户的 Release Notes 和一份面向内部团队的技术变更日志。"

## 如何用 Glob 快速找文件

你不需要知道文件在哪个目录，直接在左侧搜索框用 Glob 模式搜。

常用模式：

| 你想找什么 | 输入 |
|---|---|
| 所有接口定义文件 | `*.api.ts` |
| 所有控制器 | `*Controller*` |
| 所有测试文件 | `*.test.ts` 或 `*.spec.ts` |
| 所有配置文件 | `*.config.*` |
| 某个目录下的所有文件 | `src/api/**` |
| 所有页面组件 | `src/pages/**/*.tsx` |
| 包含"order"的文件 | `*order*` |

**实用场景**：想评估某个功能涉及多少文件时，先用 Glob 搜出来，看看数量和分布，大概就能判断改动范围大不大。

## 常用 Skill 模板

在 Agent 面板的 **Skills** 里创建以下模板，以后一键插入：

### Skill 1：解释代码功能（`explain-code.md`）

```markdown
# 解释代码功能

请用产品经理或非技术人员能看懂的语言，解释我提供的代码实现了什么功能。

说明要包括：
1. **这段代码做什么**：用一两句话总结
2. **完整流程**：按步骤描述，避免技术术语
3. **输入和输出**：接收什么数据，产出什么结果
4. **限制和边界**：什么情况下会失败或有限制
5. **对用户的影响**：这段逻辑对用户体验有什么直接影响

不需要修改代码，只需要解释。
```

### Skill 2：评估改动影响（`assess-impact.md`）

```markdown
# 评估改动影响

我会提供一段代码或改动描述，请评估这个改动的影响范围和工作量。

输出内容：
1. **改动摘要**：一句话说清楚改的是什么
2. **需要改动的文件**：列出文件路径，每个文件说明改什么
3. **风险点**：有哪些可能出问题的地方
4. **依赖关系**：这个改动是否依赖其他功能先完成
5. **工作量估计**：小（1 天以内）/ 中（1-3 天）/ 大（3 天以上），说明理由

面向非技术读者，不要假设读者懂代码。
```

### Skill 3：生成 Changelog（`gen-changelog.md`）

```markdown
# 生成 Changelog

根据我提供的代码改动，生成两份文档：

**用户版 Release Notes**（对外，用户语言）：
- 新功能：用户能用到什么新东西
- 优化：哪些体验变好了
- 修复：修了哪些用户遇到的问题
- 不写纯技术改动

**内部技术变更日志**（对内，简洁技术语言）：
- 按模块列改动
- 标注 Breaking Changes
- 标注需要运维配合的部分（比如数据库迁移、配置变更）

不需要修改代码，只生成文档。
```
