# CodeExtractor 测试人员使用指南

这份文档面向 QA 和测试同学，帮你用 CodeExtractor 的 Agent 模式提升测试效率——不需要自己写代码，但能用 AI 快速看懂改动、生成测试用例、分析影响范围。

---

## 测试人员能用 Agent 做什么

你不需要能改代码，才能用好 Agent。Agent 对测试人员最有价值的几个场景：

- **看懂代码改动**：把 diff 或改动文件交给 Agent，让它用人话解释改了什么、为什么改
- **生成测试用例**：给一段功能代码，让 Agent 列出测试 case（包括边界场景）
- **分析影响范围**：提交了一个 PR，让 Agent 告诉你哪些功能可能受影响，重点测哪里
- **复现 bug 原因分析**：把错误日志和相关代码给 Agent，让它解释为什么会出这个错
- **生成回归测试清单**：选中改动的文件，让 Agent 输出一份结构化的回归检查清单

**重要提示**：测试场景下，强烈建议使用**仅规划模式**，Agent 只分析不修改任何文件，你只需要读输出就好。

## 典型场景一：看 PR 改动，分析影响范围

**背景**：开发提了一个 PR，改动涉及用户鉴权模块，你需要知道要测哪些地方。

**操作步骤：**

1. 在 CodeExtractor 左侧面板，选中 PR 里改动的文件（可以从 PR 的 Files Changed 页面对照着选）
2. 切换到 Agent 面板，权限模式选**仅规划**
3. 在任务输入框里写：

   > "这些文件是本次 PR 的改动。请分析：1）改动了哪些核心逻辑；2）哪些已有功能可能受到影响；3）需要重点验证的测试点有哪些，按优先级排列。不需要修改任何代码，只给我分析报告。"

4. 按 Cmd+Enter 运行，等 Agent 完成
5. 把输出复制到 Jira 或测试用例管理工具

**输出示例（Agent 会给类似这样的报告）：**

```
影响范围分析：

核心改动：
- 修改了 JWT token 的过期时间逻辑（从固定 24h 改为动态配置）
- 新增了 refresh token 的自动续期机制

可能受影响的功能：
- 登录/登出流程（高优先级）
- 记住我功能（高优先级）
- 跨 tab 登录状态同步（中优先级）
- 移动端 token 同步（中优先级）
- 接口鉴权失败的错误提示（低优先级）

建议测试点：
1. 正常登录后 token 过期，是否自动续期
2. 手动登出后 refresh token 是否失效
...
```

## 典型场景二：给一段代码，生成测试用例

**背景**：开发新写了一个工具函数，还没有测试，你想快速生成一份测试用例清单。

**操作步骤：**

1. 在左侧面板选中要测试的文件（比如 `src/utils/validators.ts`）
2. Agent 面板，权限模式选**仅规划**
3. 输入：

   > "请为这个文件里的所有函数生成完整的测试用例清单，格式如下：
   >
   > 函数名：xxx
   > - 正常路径：...
   > - 边界 case：...（空值、最大值、最小值、特殊字符等）
   > - 异常路径：...（非法输入、缺少必填字段等）
   >
   > 不需要写代码，给我可以直接在测试用例系统里录入的文字描述。"

4. 输出可以直接复制进 TestRail、Jira、或 Excel

**技巧**：如果函数依赖外部接口或数据库，加一句"假设外部依赖已正常工作，只关注函数本身的输入输出"，避免 Agent 生成一堆 mock 相关的用例。

## 典型场景三：复现 bug，让 Agent 分析原因

**背景**：线上报了一个错误，你有错误日志，想搞清楚是什么原因，方便跟开发沟通。

**操作步骤：**

1. 选中报错相关的文件（从错误栈里找文件名）
2. Agent 面板，权限模式选**仅规划**
3. 把错误信息粘贴到任务输入框里，加上说明：

   > "线上出现了以下报错：
   >
   > ```
   > TypeError: Cannot read properties of undefined (reading 'userId')
   >   at OrderService.createOrder (src/services/order.ts:45)
   >   at POST /api/orders (src/api/orders.ts:23)
   > ```
   >
   > 我已选中相关文件。请帮我：
   > 1. 解释这个错误是怎么发生的，用非技术语言
   > 2. 触发这个 bug 的具体操作路径是什么
   > 3. 哪些情况下会出现，哪些情况下不会
   > 4. 这是偶发还是必现，为什么
   >
   > 不需要修复代码，只需要分析。"

**输出你可以用来：**

- 在 Jira 里写清晰的 bug 复现步骤
- 跟开发对齐根本原因，避免来回返工
- 判断这个 bug 的严重程度和优先级

## 典型场景四：回归测试，生成检查清单

**背景**：这次版本改了不少东西，你需要一份系统的回归测试清单，覆盖所有可能的影响点。

**操作步骤：**

1. 在左侧面板选中**本次版本所有改动的文件**（可以从 git 记录或 PR 列表里整理）
2. 在 Config 标签选 XML 或 Markdown 格式（让 Agent 看得更清楚）
3. Agent 面板，权限模式选**仅规划**
4. 输入：

   > "这些是本次迭代改动的所有文件。请输出一份完整的回归测试检查清单，格式要求：
   >
   > 按功能模块分组，每个测试点说明：
   > - 要验证什么
   > - 前置条件
   > - 预期结果
   > - 风险等级（高/中/低）
   >
   > 特别关注：改动涉及的数据流、用户可见的行为变化、以及与其他模块的集成点。"

**技巧**：如果版本比较大，可以先让 Agent 输出一个"高风险改动摘要"，你优先测高风险，时间不够的话低风险可以快速过。

---

## 推荐：只用"仅规划"模式

作为测试人员，你大多数情况不需要 Agent 修改任何代码，只需要它的**分析和输出**。

**仅规划模式**的好处：

- Agent 不会碰任何文件，零风险
- 你可以把输出直接复制使用，不需要懂代码
- 适合在任何机器上运行，不需要本地有完整开发环境

在 Agent 面板顶部，权限模式选择**仅规划**，然后正常运行即可。

## 如何把 Agent 输出复制到其他工具

### 复制全部输出

Agent 完成后，在输出区域右上角有**复制**按钮，一键复制全部 Markdown 文本。

### 复制 diff 内容

如果你想把代码改动的 diff 贴到测试报告里：
1. 点开 diff 查看器
2. 展开你关心的文件
3. 选中文字手动复制，或使用文件旁的复制图标

### 整理后导入工具

Agent 输出的是 Markdown 格式，大多数工具都支持：

- **Jira**：直接粘贴，Jira 编辑器支持 Markdown（富文本模式下）
- **Confluence**：粘贴后用"Markdown 宏"渲染，或手动整理
- **TestRail / PractiTest**：把测试 case 逐条复制到对应字段
- **飞书/钉钉文档**：直接粘贴，支持 Markdown 渲染
- **Excel**：把每个 case 整理成一行，粘贴即可

## 常用 Skill 模板

在 Agent 面板里点击 **Skills**，可以创建以下模板，以后一键插入：

### Skill 1：分析影响范围（`impact-analysis.md`）

```markdown
# 分析影响范围

请分析我选中的代码改动，输出影响范围报告：

1. **改动摘要**：用一句话概括本次改动做了什么
2. **直接影响**：直接被修改的功能点列表
3. **间接影响**：可能被连带影响的其他功能（考虑调用关系、共享状态、数据库）
4. **用户可见变化**：哪些改动用户能感知到
5. **建议测试优先级**：高/中/低，并说明原因

输出格式要清晰，可以直接粘贴到测试计划里。不要修改任何代码。
```

### Skill 2：生成测试用例（`gen-test-cases.md`）

```markdown
# 生成测试用例

请为我提供的代码生成测试用例清单，不需要写代码，只需要文字描述。

每个功能点包含：
- **正常路径**：输入合法时的预期行为
- **边界 case**：空值、0、最大值、最小值、超长字符串、特殊字符
- **异常路径**：非法输入、权限不足、依赖服务不可用
- **并发场景**：如果涉及并发或多用户操作

格式：可以直接录入测试用例管理系统的条目。
不要修改任何代码。
```

### Skill 3：识别边界场景（`find-edge-cases.md`）

```markdown
# 识别边界场景

仔细分析我提供的代码，专门找出容易被忽略的边界场景和潜在风险点：

1. **数据边界**：空值、null、undefined、零值、负数、超大数、特殊字符
2. **状态边界**：初始状态、中间态、终态之间的转换
3. **并发风险**：多个请求同时进来会发生什么
4. **依赖风险**：如果外部依赖（API、数据库）响应异常会怎样
5. **权限边界**：不同角色/权限的用户操作同一个功能会有什么不同

每个场景说明：为什么这是个风险、可能的后果是什么。不要修改代码。
```

### Skill 4：生成回归检查清单（`regression-checklist.md`）

```markdown
# 生成回归检查清单

根据我提供的代码改动，生成一份回归测试检查清单。

格式要求：
- 按功能模块分组
- 每条测试项包含：测试内容、前置条件、预期结果、风险等级（高/中/低）
- 重点关注：改动的核心逻辑、与其他模块的集成点、数据一致性

输出可以直接用于测试执行追踪。不要修改任何代码，只输出检查清单。
```
