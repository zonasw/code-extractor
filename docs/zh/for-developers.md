# 开发者使用指南

这份文档面向日常写代码的你，帮你把 CodeExtractor 真正融入开发工作流——不只是"把文件复制给 AI"，而是用 Agent 模式高效完成新增功能、修 Bug、重构、写测试等真实任务。

---

## 常见开发任务示例

### 新增功能

把相关模块选中（接口定义、已有的类似功能、类型文件），然后直接描述需求：

```
参照 src/api/posts.ts 的写法，在 src/api/comments.ts 里新增一个
deleteComment(id: string) 方法，包含完整的错误处理，并更新
src/types/comment.ts 中的对应类型定义。
```

::: tip
新功能开发推荐用**目录模式**——让 Agent 自己去读相关文件，省得你费心挑选。
:::

### 修 Bug

把报错的文件和完整错误日志一起提供给 Agent，上下文越完整，诊断越准确：

```
运行时出现以下报错：

TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (src/components/UserList.tsx:42:18)
    at renderWithHooks (react-dom.development.js:14985:18)

这个错误在切换到"团队视图"时必现。帮我找出根本原因并修复，
不要引入新的 async 状态管理依赖。
```

::: tip
修已知 Bug 推荐用**内联模式**，把报错的组件和它依赖的 hook 直接选中，Agent 能拿到完整上下文。
:::

### 重构模块

重构最怕 Agent 改太多。先用**仅规划**模式看方案，确认没问题再执行：

```
把 src/components/DataTable/ 里所有直接操作 DOM 的地方（querySelector、
innerHTML 等）改成用 React ref 实现。其他逻辑保持不变。

先给我一个详细的改动计划，列出每个文件需要改哪些地方，不要直接修改文件。
```

看完计划觉得方向对了，切换到"接受编辑"模式，用同样的 prompt 重新执行。

### 给已有代码写单测

选中要测试的文件，明确说明你用的测试框架和期望的测试范围：

```
为 src/utils/formatters.ts 里所有导出的函数写 Vitest 单元测试。

覆盖范围：
- 正常路径（典型输入）
- 边界条件（空字符串、null、undefined、超长字符串）
- 异常输入（类型错误、非法格式）

测试文件放在 src/utils/__tests__/formatters.test.ts。
测试描述用中文，不要写多余的注释。
```

## 如何选择上下文模式

### 内联模式

把你勾选的文件内容**直接嵌入 prompt**，Agent 拿到的就是完整的文件文本。

**适合：**
- 任务范围明确，相关文件就那几个
- 你希望 Agent 严格聚焦，不要到处乱跑
- 文件总 token 在 50K 以内
- 对特定代码行有针对性的修改

**不适合：**
- 文件太多、太大（超 token 限制）
- 你不确定哪些文件相关

### 目录模式

只把目录路径传给 Agent，让它用 Read / Glob / Grep 工具自己去探索。

**适合：**
- 大型项目，你不确定哪些文件相关
- 任务涉及多个模块（比如"全局替换某个 API 的调用方式"）
- 探索性任务，希望 Agent 自己分析依赖链

**不适合：**
- 任务已经很明确、文件很少的情况（反而浪费 token）

### 决策表

| 任务类型 | 文件数量 | Token 预算 | 推荐模式 |
|---|---|---|---|
| 修一个具体的 Bug（文件已知） | 1-3 个 | < 30K | 内联 |
| 给某个函数写单测 | 1-2 个 | < 20K | 内联 |
| 新增功能，涉及多个模块 | 3 个以上 | 不确定 | 目录 |
| 重构整个模块 | 多个 | 较大 | 目录 |
| 分析全局影响范围 | 全项目 | 大 | 目录 |
| 代码审查、安全扫描 | 全项目 | 大 | 目录 |

::: warning
选文件太多：噪音变多，Agent 容易走偏，token 费用也高。
选文件太少：Agent 缺乏上下文，给出的方案容易和现有代码风格不符。
:::

## 权限模式详解

### 仅规划（Plan Only）— 最推荐入门方式

Agent 只分析和输出改动计划，**不修改任何文件**。你能看到：
- 哪些文件会被改
- 每个文件改什么、改哪一行
- 改动的先后顺序和逻辑依据

**最安全的工作流：**

1. 用**仅规划**模式提交任务 → 查看输出的改动方案
2. 如有问题，继续追问澄清（多轮对话）
3. 确认方案没问题 → 切换到**接受编辑**模式 → 用同一个 prompt 重新执行
4. 任务完成后查看 diff → 满意则 commit，不满意则 revert

::: tip 重构和大改动必用仅规划
重构涉及文件多、改动面广，先看计划再执行能避免"Agent 自作聪明改了不该改的地方"。
:::

### 接受编辑（Accept Edits）

Agent 可以直接读写文件，无需每步确认。适合任务明确、改动范围清晰的场景。运行后检查 diff，不满意就 revert。

### 默认（Default）

对敏感操作（写文件、执行命令）会暂停等待你确认。适合你对 Agent 的行为还不太有把握的情况。对大多数日常任务来说，默认模式已经足够安全。

### 跳过权限（Skip Permissions）

所有操作自动批准，包括执行任意 Bash 命令。

::: warning
只在你完全信任 prompt 内容时使用，例如跑已知的 lint 修复或格式化命令。不要在有副作用的任务上使用跳过权限模式。
:::

## 多轮对话策略

### 什么时候继续上一个会话（--resume）

- 上一个任务做了一半，想继续完成
- 想在上一步结果的基础上追加需求
- Agent 的第一次实现有问题，想让它在原有上下文里自行修正

在 Agent 面板的**会话历史**侧边栏找到对应的 session，点击继续即可（内部使用 `--resume` 标志）。

### 什么时候开新会话

- 任务性质完全不同（比如刚写完一个功能，下一个要修另一个模块的 Bug）
- 上一轮的改动已经 commit，开始全新的需求
- 历史上下文太长、太杂，可能干扰 Agent 的判断

**原则：** 一个会话聚焦一件事。塞太多不同任务的历史上下文会让 Agent 混淆。

### 示例：一个功能分 3 轮完成

**第 1 轮 — 搭骨架**

```
在 src/features/notifications/ 下新建通知模块的基础结构：
NotificationService 类、对应的类型定义、以及一个空的 React Provider。
先不实现具体逻辑。
```

**第 2 轮 — 继续上个会话，实现核心逻辑**

```
继续上一步。现在给 NotificationService 实现以下方法：
add(notification)、dismiss(id)、dismissAll()。
每个方法都要有完整的 TypeScript 类型。
```

**第 3 轮 — 继续，写测试**

```
继续上一步。为 NotificationService 的三个方法写 Vitest 单元测试，
测试文件放在 src/features/notifications/__tests__/。
```

### 追问技巧

Agent 给的方案不完全对时，不需要重来，直接追问：

```
你刚才的实现在并发场景下有问题：多个请求同时触发时，
userCache 会被覆盖。请修改这部分，其他已完成的部分不动。
```

### 会话历史的价值

会话历史不只是"继续"的入口，还能帮你：
- 回顾某次 Agent 任务的完整改动记录
- 对比"上次的方案"和"这次的方案"的差异
- 在团队里分享某次 Agent 工作流的快照

## 开发者推荐 Skills

Skills 是存放在 `~/.claude/commands/*.md` 的 Markdown 文件，在 Agent 面板里一键插入到任务输入框。把以下文件放到对应路径即可使用。

### code-review.md

```markdown
审查我提供的代码，检查以下四个维度，输出带严重程度分级的检查清单：

1. **潜在 Bug**：逻辑错误、空值/undefined 未处理、边界 case 遗漏
2. **性能问题**：不必要的重复计算、内存泄漏、N+1 查询、大数组操作
3. **安全漏洞**：SQL 注入、XSS、敏感信息泄露、不安全的依赖
4. **代码风格一致性**：命名规范、函数职责单一、重复代码

每个问题格式：
- 严重程度：🔴 严重 / 🟡 建议 / 🟢 优化点
- 问题描述（一句话）
- 文件位置（文件名 + 行号）
- 修改建议
```

### write-tests.md

```markdown
为我提供的代码编写全面的单元测试，要求：

- 覆盖所有导出的函数和方法
- 每个函数至少包含：正常路径、边界条件（空值/极值）、异常情况（错误输入/抛错）
- Mock 所有外部依赖（HTTP 请求、数据库、文件系统、时间）
- 使用项目现有的测试框架（检查 package.json 确认）
- 测试描述用中文，清晰说明"在什么情况下期望什么结果"

输出可直接运行的测试文件，不要额外解释。
```

### explain-code.md

```markdown
用简洁清晰的语言解释这段代码：

1. **它做了什么**：一句话概述功能
2. **怎么实现的**：关键步骤和数据流向（不要逐行翻译，抓主干）
3. **值得注意的设计决策**：为什么这样设计？有哪些权衡取舍？
4. **潜在的改进空间**：如果有的话

目标读者：熟悉编程但不了解这段代码的同事。
```

### find-bugs.md

```markdown
仔细分析这段代码，找出所有潜在问题：

- 空指针 / undefined 访问
- 竞态条件（async/await、Promise、并发状态更新）
- 数组越界 / 迭代器边界错误
- 未处理的错误情况（try/catch 缺失、Promise rejection 未捕获）
- 类型不匹配或隐式类型转换
- 资源泄漏（未关闭的连接、未清理的定时器）

每个问题给出：问题描述 + 触发条件 + 修复建议。
按严重程度排序，最危险的放最前面。
```

## Git 工作流集成

### 标准工作流

**第一步：在功能分支上工作**

```bash
git checkout -b feature/my-new-feature
```

Agent 面板顶部会显示当前 git 分支，每次跑 Agent 前确认分支名称正确。

**第二步：用"仅规划"审阅方案**

先用仅规划模式提交任务，看 Agent 输出的改动计划。确认方向正确后，再切换到"接受编辑"重新执行。

**第三步：查看 Diff 查看器**

任务完成后，逐文件检查 diff，重点关注：
- Agent 有没有动它不该碰的文件
- 逻辑是否符合预期
- 有没有留下 `console.log`、`TODO` 或调试代码

**第四步：提交或回滚**

- 改动符合预期：`git add` + `git commit`
- 不满意：点击 **Revert** 执行 `git restore .`，修改 prompt 后重跑

**第五步：PR 前自审**

提 PR 前，用 `code-review` Skill 让 Agent 自审一遍，把明显问题在 code review 前修掉。

### 未提交改动警告

Agent 面板检测到工作区有未提交改动时，会在顶部显示警告。这意味着 Agent 运行后的 diff 里会混入你之前的改动，难以分辨。

::: warning 建议
运行 Agent 前，先 `git stash` 或 `git commit` 暂存现有改动，保持工作区干净。
:::

### 注意事项

- 不要在 `main` 或 `master` 分支上直接跑有写权限的 Agent
- Revert（`git restore .`）会丢弃所有未提交改动，包括 Agent 运行前你自己的修改——如果你有想保留的内容，先 commit

## 选择预设的妙用

选择预设（Selection Presets）可以把常用的文件组合保存下来，一键切换，告别每次手动勾选。

### 创建预设

1. 在左侧面板勾选目标文件
2. 打开 **配置** 标签
3. 在"选择预设"区域点击 **保存当前选择**，输入预设名称
4. 下次从下拉列表一键载入

### 推荐预设方案

**前端组件预设**

用 Glob 选中：`src/components/**/*.tsx`、`src/hooks/**`、`src/styles/**`

用途：UI Bug 修复、新增组件、样式调整任务

**API 层预设**

用 Glob 选中：`src/api/**`、`src/types/**`、`src/middleware/**`

用途：新增接口、数据验证、错误处理逻辑

**测试文件预设**

用 Glob 选中：`**/*.test.ts`、`**/*.spec.ts`、`**/*.test.tsx`

用途：查看测试覆盖情况、批量补充测试用例

**配置文件预设**

用 Glob 选中：`*.config.*`、`tsconfig.json`、`package.json`

用途：排查依赖问题、调整构建配置、环境变量审查

::: info
在 Glob 搜索栏输入模式后，目录树会过滤为匹配文件，勾选根节点可以一次性全选所有匹配的文件。
:::
