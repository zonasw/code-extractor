# 产品经理使用指南

你不需要会写代码，就能用 CodeExtractor 理解代码库、评估需求工作量、分析技术债，甚至生成发布说明。这份指南从产品视角出发，告诉你每个场景该怎么做。

---

## 两种用法：选一个适合你的

### 简单模式 — 提取代码，粘贴到 Claude.ai（无需 API Key）

1. 打开 CodeExtractor，加载项目目录
2. 用 Glob 找到相关文件，勾选它们
3. 点 **复制**（`Ctrl+Shift+C`）
4. 打开 [claude.ai](https://claude.ai)，粘贴，提问

这个模式不需要配置任何东西，也不需要技术背景。代码内容就像一份文档，你把它交给 Claude，然后用自然语言提问。

::: tip 这个模式适合你，如果……
- 你只是偶尔需要了解某个功能的实现
- 你用的是公司的 Claude.ai 账号，不方便配置 API Key
- 你只需要阅读和理解代码，不需要 AI 去修改代码
:::

### 进阶模式 — Agent 模式（需要 API Key）

打开 **配置** 标签填入 API Key，然后在 **Agent** 标签里直接输入任务，让 AI 自动分析整个代码库并输出结构化报告。

适合需要 AI 主动探索大型代码库、生成详细分析报告的场景。

::: info 获取 API Key
前往 [console.anthropic.com](https://console.anthropic.com) 注册并生成 API Key（格式为 `sk-ant-...`）。如果公司有 OpenRouter 账号，也可以用 OpenRouter Key 配合 Base URL `https://openrouter.ai/api/v1` 使用。
:::

## 场景一：理解某个功能是怎么实现的

**"这个登录流程到底怎么做的？有哪些边界情况？"**

**步骤 1 — 用 Glob 找相关文件**

在左侧面板的搜索栏输入 Glob 模式，快速定位相关模块：

```
**/auth/**
```

或者更精确：

```
**/*login*
**/*signin*
```

**步骤 2 — 勾选文件，复制或提交 Agent 任务**

```
用非技术语言解释这段代码实现的功能。

我想了解：
1. 从用户视角看，这个功能的完整流程是什么？
2. 用户会遇到哪些情况？（正常登录成功、各种失败情况分别是什么）
3. 有哪些边界条件可能影响用户体验？（比如：账号被锁、密码过期、并发登录）
4. 这个实现有没有已知的限制或潜在问题？

请避免使用技术术语，用产品语言描述。
```

::: tip
在简单模式下，把 token 控制在 50K 以内效果最好。太多文件会稀释 Claude 的注意力。
:::

## 场景二：评估新需求的工作量

**"我想加个 SSO 登录，大概要改多少东西？"**

**步骤 1 — 选中会受影响的模块**

根据你对功能的理解，选中可能需要改动的区域。不确定的话，用目录模式让 Agent 自己探索：

```
**/auth/**
**/user/**
**/config/**
```

**步骤 2 — 提交工作量评估任务**

```
我们计划新增 SSO（单点登录）功能，支持企业用 SAML 或 OIDC 协议接入。

基于当前代码库，请帮我评估：

1. **需要改动的文件**：列出所有受影响的文件（按改动量大小排序）
2. **新增内容**：需要新建哪些模块或文件
3. **工作量拆分**：按功能点拆分，每项给出工作量估算（S=1-2天 / M=3-5天 / L=1-2周）
4. **技术风险点**：哪些改动可能引发兼容性问题或需要额外测试
5. **推荐的实现顺序**：先做什么，后做什么

我需要这份分析来准备下周的需求评审，请用非技术人员能理解的语言描述风险和依赖。
```

**步骤 3 — 进一步追问**

如果有不明白的地方，继续问：

```
你提到"认证中间件需要重构"，能解释一下这对用户会有什么影响吗？
会不会有现有用户的登录受到影响？
```

## 场景三：技术债分析报告

**"我需要给 CTO 一份技术债现状报告，优先级最高的几项是什么？"**

**步骤 1 — 用目录模式选择整个 src/**

在左侧面板加载项目目录后，切到 **Agent** 标签，选择 **目录模式**（让 Agent 自己探索代码库，而不是手动选文件）。

**步骤 2 — 提交分析任务**

```
分析这个代码库，找出前 5 大技术债问题。

对每个问题，输出：
- **问题描述**：是什么，在哪里（文件或模块）
- **业务风险**：如果不处理，会有什么影响？（用产品语言：影响哪些用户功能、有没有安全风险）
- **修复工作量**：S（1-3天）/ M（1-2周）/ L（1个月以上）
- **修复优先级建议**：现在就要修 / 下个季度 / 长期规划

最后给一个总体结论：这个代码库的技术健康度如何？哪个方向最值得投入？

输出为 Markdown 格式，适合直接作为技术债评审文档。
```

::: tip
这类分析任务用 **Agent 模式 + 目录模式** 效果最好，因为 Agent 可以自主探索整个代码库，找出你可能不知道问题在哪里的隐患。
:::

## 场景四：生成 Release Notes / Changelog

**"这次发布改了什么？帮我写面向用户的发布说明。"**

### 方法 A：选中改动文件

在左侧面板用 Glob 选中本次发版改动的文件，然后：

```
为这些代码改动撰写面向用户的发布说明（Release Notes）。

要求：
- 聚焦"用户现在能做什么"或"用什么问题被修复了"，不要技术细节
- 按类别分组：新功能、优化改进、Bug 修复
- 每条说明一句话，简洁易懂
- 避免技术术语（不要说"重构了 API 层"，要说"加载速度提升了约 30%"）
- 最后加一个"升级注意事项"——用户是否需要做任何操作（如清除缓存、重新授权）

目标读者：普通用户，不懂技术。
```

### 方法 B：粘贴 git diff

如果你能拿到 git diff（找开发要），直接粘贴到任务输入框：

```
以下是本次版本的代码变更（git diff）：

[粘贴 git diff 内容]

请将以上技术变更转化为面向用户的发布说明，格式同上。
```

::: info
向开发同事要 git diff 的命令：`git diff v1.2.0..v1.3.0`（把版本号替换为实际值）。
:::

## 产品经理 Glob 速查表

在左侧面板的搜索栏输入以下模式，快速定位常用模块：

| Glob 模式 | 找到什么 | 适合什么场景 |
|---|---|---|
| `**/api/**` | 所有 API 接口文件 | 了解有哪些接口，评估 API 改动影响 |
| `**/*controller*` | 所有控制器文件 | 了解业务逻辑入口 |
| `**/*service*` | 所有服务层文件 | 了解核心业务逻辑 |
| `**/*.test.*` | 所有测试文件 | 看测试覆盖情况，了解开发质量 |
| `**/i18n/**` 或 `**/locales/**` | 所有翻译文件 | 评估国际化/多语言需求工作量 |
| `**/auth/**` 或 `**/*auth*` | 所有认证相关文件 | 安全需求、登录流程分析 |
| `**/payment/**` | 所有支付相关文件 | 支付流程、计费逻辑 |
| `**/*config*` | 所有配置文件 | 了解系统配置和环境变量 |
| `**/types/**` 或 `**/*.d.ts` | 所有类型定义 | 了解数据模型和数据结构 |

::: tip 组合使用
你可以多次搜索不同模式，每次勾选匹配的文件，最终汇总成一个覆盖多个模块的选择。记得用"选择预设"保存常用组合！
:::

## 产品经理推荐 Skills

把以下文件放到 `~/.claude/commands/` 目录，在 Agent 面板 Skills 菜单里一键使用。

### estimate-effort.md

```markdown
基于我提供的代码库现状，评估以下需求的开发工作量：

[在这里描述你的需求]

请输出：
1. 受影响的模块列表（简要说明每个模块要做什么）
2. 工作量拆分表（功能点 | 工作量 S/M/L | 说明）
3. 技术风险（哪些地方可能超出估算，为什么）
4. 推荐的实现顺序（有依赖关系的功能先后顺序）
5. MVP 建议（如果要先做最小可用版本，哪些是必须的，哪些可以后做）

用产品语言描述，避免技术术语。
```

### tech-debt-report.md

```markdown
分析这个代码库，生成一份技术债评估报告，用于产品-技术对齐会议。

报告包含：
1. **技术债全景**：主要问题领域和规模（不超过 5 项）
2. **业务影响评估**：每项技术债如果不处理，会对哪些用户功能或业务目标造成风险
3. **修复优先级矩阵**：按"业务风险 × 修复成本"排序，给出推荐处理顺序
4. **短期 / 长期行动建议**：哪些应该进下个 Sprint，哪些放季度规划，哪些可以忽略

用非技术人员能理解的语言，结论要有明确的行动方向。
```

### release-notes.md

```markdown
将我提供的代码变更转化为面向用户的发布说明（Release Notes）。

格式要求：
- **新功能** 🆕：用户现在能做什么新事情
- **优化改进** ✨：原有功能有哪些提升（性能、体验、稳定性）
- **Bug 修复** 🐛：修复了哪些影响用户的问题
- **升级注意** ⚠️：用户需要做任何操作吗（重新登录、清缓存、重新授权）

写作原则：
- 每条一句话，以动词开头（"支持..."、"修复了..."、"优化了..."）
- 不要技术细节，描述用户感知到的变化
- 目标读者是普通用户，不是开发者
```
