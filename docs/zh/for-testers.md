# 测试人员使用指南

CodeExtractor 让测试人员能够快速理解代码改动的影响范围，生成有结构的测试用例，分析 Bug 根因，并生成回归测试清单——不需要自己读代码，让 AI 帮你做分析，你来判断结果。最重要的一点：**测试人员应始终使用"仅规划"模式**，确保 AI 只做分析，不修改任何文件。

---

## 场景一：分析一个 PR 的影响范围

拿到一个 PR 需要评估测试范围时，不需要逐行读 diff——让 Agent 帮你梳理。

**步骤 1 — 加载仓库，找到改动文件**

在左侧面板加载项目目录，然后在搜索栏用 Glob 快速定位改动的文件。例如，这次 PR 改动了支付模块：

```
**/payment/**
```

或者根据具体文件名搜索：

```
**/checkout*
```

**步骤 2 — 切到 Agent 标签，选择"仅规划"模式**

在 Agent 面板的权限模式选择器中选择 **仅规划**，这样 Agent 只会分析，不会修改任何文件。

**步骤 3 — 提交分析任务**

```
请分析以下代码改动（见已选文件），从测试人员视角输出：

1. 受影响的功能列表（按模块分组）
2. 潜在的回归风险点（哪些原有功能可能被这次改动意外影响）
3. 建议的测试用例方向（不需要写详细步骤，给出测试场景标题即可）
4. 需要重点关注的边界条件

输出格式：分组的 Markdown 列表，方便直接复制到测试计划文档。
```

::: tip
如果你有 PR 的 git diff 文本，可以直接粘贴到任务输入框里，配合"目录模式"让 Agent 对照完整代码库进行分析。
:::

## 场景二：生成测试用例

需要为某个功能或模块生成系统性的测试用例时：

**步骤 1 — 选中要测试的文件**

在左侧面板勾选目标文件。如果要测试用户注册流程，选中 `src/api/auth.ts`、`src/components/RegisterForm.tsx` 等相关文件。

**步骤 2 — 提交任务**

```
分析我选中的代码，为其核心功能生成完整的测试用例。

输出格式为表格，每行一个测试用例，包含以下列：
| 用例 ID | 功能模块 | 测试描述 | 前置条件 | 测试步骤 | 预期结果 | 优先级（P0/P1/P2） |

要求：
- 覆盖正常流程（Happy Path）
- 覆盖异常流程（输入非法值、网络失败、权限不足等）
- 覆盖边界条件（最大长度、最小值、空值等）
- P0 用例应包含核心业务路径上的所有场景
```

**步骤 3 — 复制到你的测试管理工具**

把 Agent 输出的表格直接复制，粘贴到 Jira、飞书、Confluence 或 TestRail（参见[把输出导入你的工具](#把输出导入你的工具)）。

## 场景三：Bug 根因分析

收到一个 Bug 报告，需要快速定位根因：

**步骤 1 — 选中相关文件**

根据 Bug 描述，在左侧面板选中涉及的功能模块文件。如果不确定哪些文件相关，用目录模式让 Agent 自己找。

**步骤 2 — 把错误日志粘贴到任务输入框**

```
收到以下线上报错，请帮我分析根本原因：

错误信息：
TypeError: Cannot read properties of null (reading 'userId')
    at OrderService.createOrder (src/services/OrderService.ts:87:32)
    at async POST /api/orders (src/routes/orders.ts:24:18)

复现路径：用户在购物车为空的情况下直接访问 /checkout 页面时触发。
浏览器：Chrome 120，移动端 Safari 也有复现。
出现频率：每天约 30 次，集中在 20:00-22:00。

请分析：
1. 根本原因在哪里（具体文件和行号）
2. 触发这个 Bug 的完整路径
3. 是否有其他类似的潜在风险点
4. 建议的修复方向（描述方案，不需要写代码）
```

::: info
把时间段、出现频率、浏览器版本等上下文信息一起提供给 Agent，有助于发现更深层的规律（比如"是不是只在特定状态下触发"）。
:::

## 场景四：回归测试检查清单

每次 Sprint 结束后，需要一份结构化的回归测试清单：

**步骤 1 — 选中这个 Sprint 改动的文件**

根据本次 Sprint 的需求范围，在左侧面板选中相关文件，或用 Glob 按模块筛选：

```
src/features/cart/**
src/features/checkout/**
src/api/payment*
```

**步骤 2 — 提交任务**

```
基于这些改动文件，生成本次 Sprint 的回归测试检查清单。

要求：
- 按功能模块分组（每组一个 ## 标题）
- 每项检查内容用复选框格式（- [ ] 描述）
- 优先列出与改动直接相关的核心路径
- 在每组末尾附上"周边影响检查"——改动可能波及到但不在本次需求范围内的区域
- 最后加一节"回归测试执行说明"，说明建议的执行顺序和优先级

输出 Markdown 格式，方便直接放入 Confluence 或飞书文档。
```

---

## 核心建议：始终用"仅规划"模式

::: warning 重要
作为测试人员，你的工作是**分析**代码，而不是修改代码。

请始终在 Agent 面板的权限模式选择器中选择 **仅规划**。这个模式从技术上保证 Agent 不会写入、编辑或删除任何文件——即使 prompt 里有类似"修复这个 Bug"的措辞，Agent 也只会输出分析和方案，不会动实际文件。
:::

如何确认你在仅规划模式下：Agent 面板顶部的模式标签显示"仅规划"，运行结果中不会出现任何 Write/Edit 工具调用卡片。

## 把输出导入你的工具

Agent 的输出是纯文本 / Markdown 格式，可以直接复制粘贴到主流工具：

**Jira**
- 复制 Agent 输出的 Markdown 表格或列表
- 在 Jira Issue 描述区域粘贴，Jira 支持 Markdown 渲染
- 或粘贴到文本框，手动转为 Jira 格式（`||` 表头）

**飞书文档 / 飞书 Wiki**
- 直接粘贴 Markdown，飞书文档支持原生 Markdown 渲染
- 表格、代码块、复选框均可正常显示

**Confluence**
- 在编辑器里使用"插入 > Markdown"功能粘贴
- 或者复制后使用 Confluence 的 Markdown 宏

**TestRail**
- 把测试用例表格的每行手动映射到 TestRail 的用例字段
- 或导出为 CSV，通过 TestRail 的批量导入功能导入

::: tip 提高输出质量的技巧
在 prompt 里明确要求输出格式（表格、清单、Markdown）会让 Agent 的输出更容易直接粘贴使用，减少手动整理的工作量。
:::

## 测试人员推荐 Skills

把以下文件放到 `~/.claude/commands/` 目录，在 Agent 面板 Skills 菜单里一键插入到任务输入框。

### test-impact.md

```markdown
从测试人员的视角分析我提供的代码改动，输出：

1. **受影响的功能模块**：按模块分组列出所有受影响的用户功能
2. **回归风险点**：哪些原有功能可能被这次改动意外影响（按风险高低排序）
3. **测试场景建议**：需要覆盖的测试场景标题列表（Happy Path + 异常 + 边界）
4. **不需要重测的范围**：明确说明哪些功能与本次改动完全无关

输出 Markdown 格式，方便直接放入测试计划文档。
```

### generate-testcases.md

```markdown
为我提供的代码生成完整的测试用例，输出为 Markdown 表格：

| 用例 ID | 功能模块 | 测试描述 | 前置条件 | 测试步骤 | 预期结果 | 优先级 |

优先级说明：
- P0：核心业务流程，每次发布必测
- P1：重要功能，版本测试需覆盖
- P2：边缘场景，有时间时测

要求：
- 每个主要功能点至少有 1 个 P0 用例
- 异常场景（网络失败、非法输入、权限不足）必须覆盖
- 每个用例的"测试步骤"尽量具体，可以直接执行
```

### root-cause-analysis.md

```markdown
我会提供一个 Bug 的错误信息和相关代码，请帮我做根因分析：

1. **根本原因**：具体是哪个文件、哪一行、什么逻辑导致了这个问题
2. **触发路径**：完整描述从用户操作到错误发生的调用链
3. **影响范围**：这个 Bug 还可能在哪些其他场景下出现？有多少用户可能受影响？
4. **临时规避方案**：在正式修复前，有没有可以告知用户的临时绕过方法
5. **修复建议**：描述修复思路（不需要写代码）

输出要简洁，结论要明确——我需要能够直接把这个分析填入 Bug 报告。
```
